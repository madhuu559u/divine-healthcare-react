Why Supabase?

  • No server to manage — Supabase is a hosted PostgreSQL database with built-in Auth, Storage, and REST API
  • Free tier — 500MB database, 1GB storage, 50,000 monthly active users, unlimited API requests
  • Row-Level Security (RLS) — database-level access control means the frontend can talk directly to the DB safely
  • Real-time capable — can add real-time updates later without architecture changes
  • Easy to switch — just change 2 env vars and re-run the SQL migration
  • Auth built-in — email/password login with session management, no custom auth server needed
  • Storage built-in — file uploads (documents, resumes, IDs) stored securely with signed URLs

Switching to a New Supabase Project — Complete Checklist

Follow ALL steps in order. Missing any step will break the application.
  1. Create new Supabase project at supabase.com/dashboard
  2. Get credentials: Go to Settings → API Keys. Copy the Project URL and anon/public key
  3. Update local .env: 
    VITE_SUPABASE_URL=https://NEW_PROJECT_ID.supabase.co
VITE_SUPABASE_ANON_KEY=eyJ...new_key...
  4. Update GitHub secrets (commands above)
  5. Run SQL Migration 001: Go to new project's SQL Editor → paste contents of supabase/migrations/001_initial_schema.sql → Run
  6. Run SQL Migration 002: Paste contents of supabase/migrations/002_fix_rls_policies.sql → Run
  7. Create storage bucket: Go to Storage → New Bucket → name: documents → Public: ON
  8. Run storage policies SQL: 
    CREATE POLICY "allow_public_upload" ON storage.objects FOR INSERT WITH CHECK (bucket_id = 'documents');
CREATE POLICY "allow_public_read" ON storage.objects FOR SELECT USING (bucket_id = 'documents');
CREATE POLICY "allow_admin_delete" ON storage.objects FOR DELETE USING (bucket_id = 'documents' AND auth.role() = 'authenticated');
CREATE POLICY "allow_admin_update" ON storage.objects FOR UPDATE USING (bucket_id = 'documents' AND auth.role() = 'authenticated');
  9. Create admin user: Auth → Users → Add User → email/password → Auto Confirm ON
  10. Redeploy: git commit --allow-empty -m "switch supabase" && git push
  11. Test: Submit a test application, check admin panel, verify documents upload

Security Best Practices

  • Never share the service_role key — only use the anon key in frontend code
  • Change admin passwords regularly via /admin/settings
  • RLS is your security layer — never disable it
  • The .env file is in .gitignore — credentials never go to GitHub
  • Supabase handles SSL, backups, and database security automatically

Customization

Adding a New Public Page

  1. Create component: src/pages/NewPage.jsx
  2. Add route in src/App.jsx: <Route path="/new-page" element={<NewPage />} />
  3. Add to navigation in src/utils/constants.js: { label: "New Page", path: "/new-page" }

Adding a Field to Job Applications

  1. Database: Run ALTER TABLE applications ADD COLUMN new_field TEXT; in Supabase SQL Editor
  2. Zustand store: Add field to src/store/useApplicationStore.js in the appropriate section + reset
  3. Form step: Add input to the relevant Step component (e.g. Step1_PersonalInfo.jsx)
  4. Zod schema: Add validation in the same Step component's schema
  5. Step9 submit: Add field mapping in Step9_Review.jsx handleSubmit
  6. Step9 review: Add <Field> display in the review section
  7. Admin display: Add field in AdminApplications.jsx detail view
